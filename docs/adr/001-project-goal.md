# ADR-001: 프로젝트 목표 — 분산 가능한 리버스 프록시 서버

## 상태
채택됨

## 배경
[frp](https://github.com/fatedier/frp)는 NAT/방화벽 뒤의 서비스를 외부에 노출하는 리버스 프록시입니다.
10년간 검증된 안정적인 도구이지만, 서버(frps)는 **단일 인스턴스**로만 동작합니다.

단일 frps의 한계:
- frps가 죽으면 모든 클라이언트 연결이 끊어짐
- 수평 확장이 불가능 — 트래픽이 늘어도 서버를 추가할 수 없음
- 서버 업데이트 시 다운타임 발생

## 결정
frps를 대체하는 **drps**(distributed reverse proxy server)를 만듭니다.

핵심 원칙:
1. **frpc 호환** — 기존 frpc 클라이언트를 수정 없이 그대로 사용
2. **HTTP 전용** — 가장 많이 쓰이는 HTTP 프록시만 지원하여 복잡도 최소화
3. **분산 가능** — 여러 drps 인스턴스가 동시에 동작할 수 있는 구조로 설계

## 개발 전략

모든 것을 한 번에 만들지 않고, 단계별로 진행합니다.

```
Phase 1: 단일 인스턴스 (frps 대체)
  - frpc와 100% 호환되는 HTTP 프록시 서버
  - 이 단계에서 프로토콜과 기능을 완전히 검증

Phase 2: 분산 확장
  - 여러 drps 인스턴스가 라우팅 정보를 공유
  - frpc는 아무 drps에나 연결 가능
  - 사용자 요청은 아무 drps로 들어와도 올바른 frpc로 전달
```

## 장단점

- **장점**: frpc 생태계를 즉시 활용, 점진적 확장, 단일 인스턴스에서도 frps 대체 가능
- **단점**: frp 프로토콜에 종속 (프로토콜 변경 불가), HTTP 외 타입 미지원
- **완화**: HTTP가 실무에서 가장 많이 사용되며, 앞단 인그레스와 조합하면 대부분의 시나리오 커버 가능

## 검토했지만 채택하지 않은 대안

| 대안 | 채택하지 않은 이유 |
|------|-------------------|
| frps를 직접 수정 (fork) | 불필요한 코드가 대량 포함, 유지보수 부담 |
| 프로토콜을 새로 설계 | frpc를 수정해야 함, 개발 비용 높음 |
| 모든 프록시 타입 동시 지원 | 복잡도가 높아 개발이 지연됨 |
| 처음부터 분산으로 시작 | 프로토콜 호환부터 검증해야 안전함 |
